## 核心场景与验收标准

### 场景 1：从空输入构造 `sdk list java`

- 用户在输入区依次选择 / 输入 `sdk` → `list` → `java`。
- 验收：
  - 内容区每一步显示当前层级标题、说明、可选下一段。
  - `currentTokens` 与 `currentNode` 与每一步同步。
  - 文案根据 locale（默认 `zh-CN`）正确切换。

### 场景 2：回退删除上一段 token

- 当前为 `sdk list java`，按 Backspace（输入区为空）。
- 验收：变为 `sdk list`，内容区显示 `sdk list` 的说明与候选；再次按变为 `sdk`。

### 场景 3：执行 `sdk list java`

- 当前为 `sdk list java`，按 Enter。
- 验收：
  - Executor 用 `bash -lc 'source ~/.sdkman/bin/sdkman-init.sh && sdk list java'`。
  - `lastCommand` / `lastResult` 写入 Session。
  - 内容区显示完整结果文本。
  - 失败时保留 stderr，不静默吞错。

### 场景 4：动态 Identifier 候选

- 当前为 `sdk install java`，进入下一段时。
- 验收：候选来自 `sdk list java` 输出的 Identifier 列表，源标记为 `dynamic`。

### 场景 5：`/fix` 上下文

- 上一条命令失败后输入 `/fix`。
- 验收：AI 输入包含 `lastCommand` 与 `lastResult.stderr`，输出建议命令或修复说明。

### 场景 6：i18n fallback

- 节点缺失 `zh-CN` 文案。
- 验收：退回到 `en-US`；两者都缺失时显示节点 ID 作为最后兜底，并在日志中告警。

## 测试义务

- 单元测试覆盖：CommandNode 加载、Session 状态迁移、CandidateProvider 静态/动态、Executor shell function 调用、Slash command parser、i18n fallback。
- 手工验证：在 macOS 与 Linux 上各跑一次"7 条 SDKMAN 命令"全流程，并把记录附在 `docs/05-验收反馈/` 验收报告。

## 不通过条件（任一即不通过）

- 任一核心场景的验收点失败。
- 测试覆盖低于核心契约的关键路径。
- Executor 静默吞错或丢失 stderr。
- i18n fallback 缺失。
