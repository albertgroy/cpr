## 1. 推荐结论（不骑墙）

- **主 provider**：**DeepSeek V3**（`deepseek-chat`，国内直连，原生 strict JSON mode，~¥1/M output token，~$0.0003 / 请求）

## 3. 多 provider 适配层接口契约草稿

放在评审报告里供 agent3 实施期参考。**不写入 `cpr/server/` 任何文件**。agent3 落地时建议放 `cpr/server/llm/base.py`，每个 provider 一个文件（`deepseek.py` / `anthropic.py` / `ollama.py`）。

```python
# cpr/server/llm/base.py（草稿，仅供评审参考）
from __future__ import annotations

from dataclasses import dataclass
from typing import Any, Literal, Protocol


# ---- 输入：来自协议 §3.1 解析后的请求字段 ----

@dataclass(frozen=True)
class ResolveRequest:
    schema_version: str         # 必须 == "1"
    client_id: str
    client_version: str
    locale: str                 # "zh-CN" | "en-US"
    tool: str
    args: list[str]             # 已脱敏（client 出站脱敏；server 不再脱敏入参）
    sub_path: list[str]
    help_text: str              # 已 64KiB 截断 + 行尾归一
    help_source: str
    help_truncated_at_size: int | None


# ---- 输出：result 形态对应协议 §2.1 result 子树 ----

CandidateKind = Literal["id", "flag", "subcommand", "enum"]


@dataclass(frozen=True)
class Candidate:
    token: str
    desc: str                   # ≤80 字符
    kind: CandidateKind


@dataclass(frozen=True)
class ResolveResult:
    summary: str
    usage: str
    candidates: list[Candidate]
    notes: list[str]
    danger: bool
    danger_reason: str | None
    exec_template: str | None           # 仅 danger=True 时给
    exec_template_args: dict[str, dict[str, str]] | None
    exec_shell_mode: Literal["direct", "bash-lc-source"] = "direct"


# ---- 错误：映射到协议 §2.1 错误码表 ----

ProviderErrorCode = Literal[
    "LLM_TIMEOUT",        # provider 侧超时 → server 504
    "LLM_PARSE_FAILED",   # provider 返回非合规 JSON / 字段缺失 → server 502
    "INVALID_TEMPLATE",   # provider 输出违反 §2.1 模板约束 → server 422（自查触发）
    "SERVER_ERROR",       # provider 网络 / 鉴权 / 5xx → server 500
]


class ProviderError(Exception):
    def __init__(self, code: ProviderErrorCode, message: str = "", *, retryable: bool = False) -> None:
        super().__init__(message)
        self.code = code
        self.message = message
        self.retryable = retryable


# ---- Provider 抽象 ----

class LLMProvider(Protocol):
    """单 provider 适配层。
    实现方约束：
    - 不得直接 raise provider SDK 的原生异常；统一转 ProviderError，code 落入上表 4 个之一。
    - resolve() 必须自查输出符合协议 §2.1：candidates[].kind ∈ 4 枚举；
      exec_template 满足 ^[a-z][a-z0-9_]*$ 占位符 + 禁用字符表 + ASCII 32-126。
      自查不过 → raise ProviderError("INVALID_TEMPLATE")。
    - timeout 由调用方传入；provider 内部不得擅自降级到更短/更长。
    - retry 由调用方控制；provider 内 SDK retry 关掉（max_retries=0 或等价）。
    - prompt_version 不在此层；由 server 路由层根据 (provider_name, prompt_template_hash, model_name) 复合生成后写回 §2.1 响应。
    """

    name: str  # "deepseek" | "anthropic" | "ollama" | ...

    def resolve(self, request: ResolveRequest, *, timeout_s: float) -> ResolveResult: ...

    def health(self, *, timeout_s: float) -> dict[str, Any]:
        """返回 {"status": "ok" | "degraded", "model": "...", "latency_ms": int}.
        失败抛 ProviderError("SERVER_ERROR")。
        不消耗 quota；server /health 路由聚合所有 provider 的 health 决定整体 status。"""
        ...


# ---- 路由器（多 provider 主备）----

@dataclass(frozen=True)
class ProviderRoute:
    primary: LLMProvider
    fallback: LLMProvider | None = None
    fallback_on: tuple[ProviderErrorCode, ...] = ("LLM_TIMEOUT", "SERVER_ERROR")


class ProviderRouter:
    """落 cpr/server/llm/router.py。
    1. 先调 primary.resolve(req, timeout_s=primary_timeout)
    2. 若抛 ProviderError 且 code ∈ fallback_on 且 fallback 非 None → 调 fallback
    3. fallback 也失败 → 直接抛最后一次的 ProviderError（server 路由层映射成 §2.1 错误响应）
    4. INVALID_TEMPLATE / LLM_PARSE_FAILED 不切换 fallback（这类是 LLM 输出问题，不是 provider 故障，切了也大概率同样错）
    """
    def __init__(self, route: ProviderRoute) -> None: ...
    def resolve(self, request: ResolveRequest, *, primary_timeout_s: float, fallback_timeout_s: float) -> tuple[ResolveResult, str]:
        """返回 (result, used_provider_name)。used_provider_name 进 server 日志便于成本归因。"""
        ...
```

**关键约定**：
- **错误码映射唯一**：provider 层错误码集合（4 个）是协议 §2.1 全表（11 个）的真子集；client-only 三条（`HELP_*` / `NETWORK_UNREACHABLE`）provider 侧不出现；`BAD_REQUEST` / `INVALID_CLIENT` / `SCHEMA_MISMATCH` / `QUOTA_EXCEEDED` 在 server 路由层处理，进不了 provider。
- **timeout 由 server 控制**：`primary_timeout_s` 建议 30s，`fallback_timeout_s` 建议 25s，整体留给 client 60s 余量内（client `server.timeout_seconds` 默认 5s 不适用 LLM 路径，需要 T-003 内调整为 60s + 在协议变更队列追一条）→ **见 §6 P2-T003-1**。
- **prompt_version 来源**：协议 §3.1 要求"server 升级 LLM / prompt / 解析逻辑后整体冷启"，所以 `prompt_version` 必须含 `(provider_name + model_id + prompt_template_hash + parser_version)` 四元复合 hash。建议 agent3 用 `f"p-{date}.{sha1(...)[:8]}"`。
- **provider 自查 INVALID_TEMPLATE**：协议 §2.1 已规定 server 自查；适配层强制实现，路由层不再做第二次自查（client 侧本就有自查兜底）。
- **fallback 触发面**：仅 `LLM_TIMEOUT` / `SERVER_ERROR` 切；`LLM_PARSE_FAILED` / `INVALID_TEMPLATE` 切了也大概率同错，浪费一次配额。

## 5. 风险与替代路径

| 风险 | 触发 | 替代路径 |
|---|---|---|
| DeepSeek 国内政策收紧 / API 突变 | 短期发生概率低，长期不可预测 | 一键切 Anthropic Haiku（适配层路由器配置变更，无需改协议、无需改 client）|
| DeepSeek 海外用户无法访问 | 海外用户用 cpr | 文档侧建议海外用户在 `~/.cpr/config` 配 `server.endpoint` 指向自部署 server；server 端配置走 Anthropic |
| 用户极度隐私敏感（请求内容含商业机密 help text）| 部分企业用户 | 自部署 server + Ollama Qwen 路径；T-003 文档侧明确"完全本地"开关 |
| LLM 输出 JSON 格式不合规率 > 5% | 模型抽风 | 适配层 INVALID_TEMPLATE / LLM_PARSE_FAILED 兜底 + server 不写缓存（协议 §3.3 已规定）；agent3 实施期需用 fixture 跑 1k 次校验合规率 |
| 单请求成本失控（用户量爆涨）| 100 用户/天上不封顶 | server quota 50/day/user 已挡；触顶用户走 §2.1 QUOTA_EXCEEDED |
| Anthropic / OpenAI 账户被风控 | 海外卡风控 | DeepSeek 单 provider 也能跑；fallback=None 是合法配置，路由器接受 |

## 6. 协议层观察（进 P2 队列，**不直接改协议文档**）

### P2-T003-1 · client `server.timeout_seconds` 默认 5s 与 LLM 路径不匹配

- 现状：`cpr/cli/api.py` 与 `~/.cpr/config` 默认 `server.timeout_seconds=5`；mock 时合理。
- 真服务现实：单次 LLM 请求 + JSON 解析 typical 8-20s（DeepSeek V3 中位 6s、Haiku 中位 4s、p99 都 > 5s）。5s 必抛 client 侧 read timeout → 用户大量见 NETWORK_UNREACHABLE。
- 建议：T-003 启动期把 client 默认 `server.timeout_seconds` 提到 **60s**；server 端 `primary_timeout_s` 30s + fallback 25s，留 5s 余量给 server 路由 + 网络。
- 是否动协议文档：协议 §1 拓扑图只写"5s server timeout"作为默认值示例；§7 字段约束也写 5s。**这是 client 默认值**，不是协议强制下限。建议改 `cpr/cli/main.py:_load_config` 默认值 + `~/.cpr/config` 模板，**不动协议文档**；如 archi 后续认为协议层应明示"LLM 路径建议 timeout"，进 P2 变更队列。
- 优先级：P2，T-003 启动期内做（agent3 第一步）。

### P2-T003-2 · `prompt_version` 复合 hash 协议未明示

- 现状：协议 §3.1 仅说"prompt_version 是 server 端 LLM prompt + 模型 + 解析逻辑的复合版本号"。
- 缺口：未明示 hash 算法。如 server 重启后 hash 漂移，将整体冷启用户缓存（轻度损失）。
- 建议：T-003 实施期定 `f"p-{YYYY-MM-DD}.{sha1(provider+model+prompt_tpl+parser_ver)[:8]}"` 算法，**写到 `docs/04-实施部署/`**，不进协议文档。
- 优先级：P2，T-003 内做。

### P3-T003-1 · 单 provider vs 一主一备的 v1 起步

- 建议 v1 上线**只跑 DeepSeek 单 provider**（`fallback=None`），把多 provider 路由层代码先写好但不启用，降低首发联调面。

- 优先级：P3，是否分两步走由 ccb_self 与 agent3 联调期决定。
