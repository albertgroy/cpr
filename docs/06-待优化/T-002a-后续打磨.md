## P2-T002a-1 · `client_id` 轮换机制

- 现状：`client_id` 首次启动写入 `~/.cpr/config`，UUID v4，无轮换。
- 影响：用户 reset 后历史配额清零；server 看到新 UUID 算新人，存在被滥用风险。
- 触发条件：自部署侧风险低（部署方决定 limit）；官方 server 上线后真正生效。
- 修复方向：考虑绑定机器指纹或邮箱二次校验；或 server 端按 IP 速率限流叠加 quota。
- 优先级：P2，T-003 启动前再决策。

## P2-T002a-2 · `prompt_version` 暴露给 client `--diag`

- 现状：协议已规定 `prompt_version` 入 cache key，server 在响应中携带；client 收到但目前只用于本地 cache。
- 影响：用户排错时无法回报当前 server prompt 版本。
- 修复方向：T-002b 实现 `cpr --diag` 子命令时把 `prompt_version` / `client_version` / `schema_version` / cache 命中率打出来。
- 优先级：P2，T-002b 内做。

## P2-T002a-3 · `result.candidates[].kind` 增加 `value`

- 现状：当前枚举 `id` | `flag` | `subcommand` | `enum`。
- 影响：自由值参数（如 `git commit -m <text>` 中的 `<text>`）只能塞 `id` 或 `enum`，语义不清。
- 修复方向：v2 加 `value` kind；v1 实现期 LLM prompt 限制只出 4 类，自由值场景文档显式说明用 `<placeholder>` 文本。
- 优先级：P2，T-003 prompt 设计时再决定是否提到 v1.x。

## P2-T002a-4 · `GET /resolve` 是否支持

- 现状：协议仅 POST `/resolve`；GET 仅 `/quota` 与 `/health`。
- 影响：风格不统一；help_text 大不适合 GET，但小命令场景可考虑。
- 修复方向：v1 不开 GET；记入待优化。

## T-002b 冒烟期实施澄清（2026-06-19）

来源：T-002b 冒烟验收（agent1，job_e863da928961；`docs/05-验收反馈/2026-06-19-T-002b-冒烟.md`）。

### P2-T002b-1 · 危险确认 EOF / 无 stdin 行为

- 现状：`cpr/cli/main.py:123` 用 `input("确认执行？(y/N) ")` 读用户输入；非交互/无 stdin 上下文（CI、`< /dev/null`、管道、被 wrap 的 shell）触发 `EOFError`，进程以 `exit=1` 抛 traceback 退出。
- 协议歧义：协议 v1 §6 + 需求 §5 明确"默认 N → 不执行"，但未规定 EOF / 无 stdin 的兜底语义。
- 实施期裁定（ccb_self，2026-06-19）：**EOF / 无 stdin 视为 N**，与 §6"默认 N"一致；不执行命令；写一行 stderr 提示（如 `非交互上下文：默认拒绝执行`）；进程 `exit=0`。
- 修复方向：T-002b 实现期在 `_maybe_execute` 用 `try/except EOFError` 包住 `input()`，捕获后按 N 处理；同时检查 `sys.stdin.isatty()`，无 tty 时不应再调 `input()`，直接走 N 分支。
- 测试义务：`tests/cli/test_more_main.py` 新增 1 条断言（`stdin = StringIO("")` 模拟 EOF，期望 exit=0、不执行 executor）。
- 优先级：P2，T-002b 修复期内做（不放 v2）。
- 协议影响：不改协议文档。如 archi 终评对此条有不同意见，回协议变更队列再说。

### P1-T002b-2 · `server.endpoint` 拼接重复 + 产品决策"endpoint 写死"

- 现象：用户实测 `cpr sdk` 输出 `not found`，`exit=3`。诊断：`cpr/cli/api.py` `ApiClient` 把传入的 endpoint 当 base URL，又拼 `/resolve` → `http://127.0.0.1:18888/resolve/resolve` → 404 → mock 返回 `{"error":{"code":"SERVER_ERROR","message":"not found"}}` → client 显示 "not found"。
- 协议口径：协议 v1 §1 拓扑图，server 同时提供 `/resolve` `/quota` `/health`，**`endpoint` 字段语义应为 base URL**。当前 agent2 的 README / 实施部署 / 默认 config 模板写成完整 `/resolve` URL，是错的。
- 用户决策（leel，2026-06-20）：**正式发布形态下 endpoint 不需要用户配置，写死在代码里**；用户不应需要手改配置文件。`~/.cpr/config` 的 `server.endpoint` 字段在自部署 / 开发期保留（用于改本地 mock 或自部署 server），但默认不下发；首次启动生成的 config 不包含 `server.endpoint`。
- 实施期裁定（ccb_self，2026-06-20）：
  - `cpr/cli/api.py`：endpoint 必须是 base URL；`resolve()` 拼 `/resolve`，`quota()` 拼 `/quota`；如果检测到 endpoint 末尾带 `/resolve` 自动 strip + 一行 stderr 提示 `(deprecated: server.endpoint 应为 base URL)`，保旧 config 不炸
  - 引入常量 `DEFAULT_ENDPOINT = "http://127.0.0.1:18888"`（开发期默认；T-003 上线时改为发布 server）；通过 `cpr/cli/main.py` 中的 `DEFAULT_ENDPOINT` 写死路径
  - 首次启动生成的 `~/.cpr/config`：**不写 `server.endpoint` 字段**（让 client 走代码默认值）；自部署用户可手动追加 `server.endpoint: http://your-server` 覆盖
  - README / `docs/04-实施部署/`：把"配置 endpoint"改为"开发/自部署可选项"；正式用法不提
- 测试义务：
  - `tests/cli/test_main_api.py` 加 2 条：(a) 旧 `endpoint: http://x/resolve` 自动 strip 后能跑通；(b) 不带 `endpoint` 字段时走 `DEFAULT_ENDPOINT`
- 优先级：P1（阻塞用户实测验收 → 阻塞 archi 终评）。
- 协议影响：不改协议文档（协议本就只说"server 提供这几个 path"，没强制 endpoint 字段语义）；只改 client 实现 + 默认 config + 用户文档。
