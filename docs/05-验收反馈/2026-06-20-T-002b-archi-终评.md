### B-2 · `client.confirm_danger: always|once|never` 配置完全未读（中）

- 现象：协议 §6 + P1-5 明确"`client.confirm_danger: always | once | never` + `--yes / -y` CLI 短路；`once` v1 仅进程内 set 不持久化"。`cpr/cli/main.py` 全文搜索 `confirm_danger` 只有 0 命中（除注释）。当前只有 `args.yes`（==`never`）能短路；`always` 与 `once` 完全未实现，配置文件里写了也是默默被忽略。
- 影响：协议字面承诺与实现不符；用户配 `confirm_danger: never` 期望全自动跑也不生效（必须每次都加 `-y`）。
- 修复方向：
  1. `_load_config` 之后从 `config["client"]["confirm_danger"]` 读出三态值（默认 `"always"`）。
  2. `_maybe_execute(tool_args, response, args.yes, confirm_mode, session_set)` 增 2 个参数。
  3. `never` → 视同 `args.yes=True`；`always` → 走标准 y/N；`once` → 进程内维护 `set[(tool, sub_path)]`，命中即跳过 prompt。
  4. `--yes / -y` 仍然短路（仅本次调用）。
- 测试义务：3 条 testcase（never→不 prompt 直接执行、once→第二次同 (tool, sub_path) 不 prompt、always→每次都 prompt）。

- 修复：由 B-1 顺手修；不需要单独改。

### B-4 · `_fixture_help` 在生产路径混入 mock 路由（中）

- 现象：`cpr/cli/main.py:88-95` 在 `find_help` 入口最先 try mock 桩——`tool.startswith("__")` 走 mock 错误码触发，`("sdk", ("install","java"))` 等 4 条 scenario tuple 也直接走桩。这是**生产二进制**的代码路径，不是只在测试里激活。意味着任何用户跑 `cpr sdk install java` **不会**真去执行 `sdk install java --help`，而是直接拿 `usage: sdk install java` 这一行假 help 当返回。
- 影响：
  - 用户实测 `cpr sdk install java` 时，client 不去抓真 help，而是把字面 `usage: sdk install java` 当 help 上传 server，server 拿这个空壳算缓存 key 并尝试 LLM 解析；mock 模式下确实能走通，但 T-003 真 LLM 接入后**会拿不到真 help_text**，缓存 key 也会与"另一台机器跑过的真 help"不一致。
  - 协议 v1 §3.1 要求 cache key 含 `help_text`（真 help 文本），现在被空壳替换。
- 修复方向：把 `_fixture_help` 抽到 `cpr/cli/_test_seam.py` 或 `tests/conftest.py` 里，仅在测试 monkeypatch 时启用；生产 `find_help` 直接 fallthrough 到 subprocess 抓 help。
- 协议影响：无。属于实现层错位，不是协议歧义。
