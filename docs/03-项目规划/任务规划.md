# 任务规划

## 阶段 0：项目命名与边界确认

目标：确认项目名称、定位和第一阶段试点范围。

任务：

1. 讨论并决定项目名称。
2. 确认产品定位：交互式命令工作台。
3. 确认第一阶段只做 SDKMAN command builder。
4. 确认是否复用 CUI 代码，还是新工程从零实现。

产出：

- 命名决策。
- MVP 范围说明。

## 阶段 1：核心模型设计

目标：定义命令树、节点、token、参数、动态候选项和多语言结构。

核心数据模型草案：

```yaml
id: sdk.list.java
tokens: [sdk, list, java]
title:
  zh-CN: 列出 Java 版本
  en-US: List Java versions
desc:
  zh-CN: 列出当前平台可安装的 Java Identifier。
  en-US: List Java identifiers available on the current platform.
children: []
execute:
  command: sdk list java
```

中间节点：

```yaml
id: sdk.list
tokens: [sdk, list]
title:
  zh-CN: list
desc:
  zh-CN: 列出 SDKMAN 支持的 candidate，或继续选择 candidate 查看版本。
children:
  - sdk.list.java
  - sdk.list.maven
  - sdk.list.gradle
```

动态参数节点：

```yaml
id: sdk.install.java.version
tokens: [sdk, install, java, "{version}"]
param:
  name: version
  source: sdk.list.java.identifiers
```

任务：

1. 定义 `CommandNode`。
2. 定义 `CommandSession`：当前 token、当前位置、上一条命令、上一条输出。
3. 定义 `CandidateProvider`：静态候选和动态候选。
4. 定义 i18n 文案格式。
5. 定义命令执行结果模型。

产出：

- 模型代码或 schema。
- SDKMAN 命令树数据。

## 阶段 2：终端 UI 原型

目标：实现“内容显示区 + 主输入区”的轻量交互。

界面结构：

```text
内容区：
  当前命令
  当前层说明
  可选下一段
  执行结果 / AI 结果

输入区：
  > sdk list _
```

交互：

- 普通输入：更新当前命令解析状态。
- 方向键：选择候选项。
- Enter：
  - 当前命令可执行时执行。
  - 当前有候选选中时追加候选 token。
- Backspace：输入区为空时删除上一段 token。
- Tab：补全当前候选。
- Esc：清空当前输入或返回上一层。

任务：

1. 选择 TUI 库：Textual / prompt_toolkit / blessed。
2. 实现基本布局。
3. 实现候选项渲染。
4. 实现点击候选项追加 token。
5. 实现键盘选择候选项。
6. 实现返回删除 token。
7. 实现执行命令并展示结果。

建议优先评估 `prompt_toolkit`：它天然适合“输入区 + 补全 + 多行说明”的命令行交互，可能比 Textual 更贴近目标体感。

产出：

- 可运行 prototype。

## 阶段 3：SDKMAN 试点

目标：完成 SDKMAN 的可用命令构造。

支持：

```text
sdk list
sdk list java
sdk list maven
sdk list gradle
sdk install java <identifier>
sdk use java <installed-version>
sdk default java <installed-version>
sdk current java
```

任务：

1. 静态定义 sdk 根命令和子命令。
2. 动态解析 `sdk list java` 的 Identifier。
3. 动态解析已安装 Java 版本。
4. 安装时提示必须使用 Identifier。
5. 对 shell function 命令统一用 `bash -lc 'source ... && sdk ...'` 执行。
6. 执行结果保存在 session，供 `/fix` 使用。

产出：

- SDKMAN command builder MVP。

## 阶段 4：Slash command AI

目标：AI 变成主输入区里的特殊命令。

支持：

```text
/ai <问题>
/explain
/fix
/help
/clear
/back
```

任务：

1. 定义 slash command parser。
2. `/explain` 读取当前命令上下文。
3. `/fix` 读取上一条失败命令和输出。
4. `/ai` 带当前上下文进行问答。
5. AI 建议命令可以被采用为当前输入。

产出：

- 命令上下文化 AI 辅助。

## 阶段 5：多语言

目标：从一开始支持多国语切换，不把中文写死在 UI 逻辑里。

任务：

1. 定义 locale 配置：`zh-CN` / `en-US`。
2. 命令节点文案支持多语言。
3. UI 固定文案支持多语言。
4. Slash command help 支持多语言。
5. 缺失语言 fallback 到 `en-US` 或 `zh-CN`。

产出：

- i18n 文案机制。

## 阶段 6：从 CUI 迁移或整合

目标：决定 CPR 与 CUI 的关系。

可选方案：

1. CPR 独立新项目。
2. CPR 成为 CUI v2。
3. CPR 作为 CUI 的一个新模式。

建议：先独立 prototype，体验确定后再考虑合并。

## 阶段 7：扩展命令域

SDKMAN 体验验证通过后，再扩展：

- git
- docker
- pyenv
- fnm/nvm
- uv/pip
- package manager: apt/brew/scoop/winget
- chsrc

每个域都按“命令树 + 说明 + 动态候选 + 执行”接入。
