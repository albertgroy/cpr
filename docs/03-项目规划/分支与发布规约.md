# 分支与发布规约

## 分支规则

- 当前项目尚未 `git init`。在仓库初始化后再正式启用本规约。
- 主分支：`main`（受保护，仅由验收通过后的合并写入）。
- 任务分支命名：`feat/T-NNN-短描述` / `fix/T-NNN-短描述` / `chore/T-NNN-短描述`。
- 每个 CCB 任务一条独立分支，单任务单 commit 优先；无关变更不要混入。

## Review Ready 条件

- 关联任务 ID 明确，链接到 `滚动执行列表.md` 行。
- 涉及契约的改动已更新 `docs/02-系统设计/` 对应契约文档。
- 单元测试与必要的手工验证已完成。
- 与其它开发对没有未协调的并发文件冲突。

## 发布证据

发布时在 `docs/04-实施部署/YYYY-MM-DD-版本发布记录.md` 记录：

- 版本号
- 包含的任务 ID 列表
- 包含的 commit 列表
- 最终验收结论与 archi 签字
- Release notes
- 回滚方案

## Tag 规则

- 仅在最终验收通过后打 tag。
- Tag 名 = 版本号。
- Tag message 引用发布记录文档路径。

## 版本号规则

格式：`YYYY.MM.<total_commit_count>`

- `YYYY`：发布日期所在年份（4 位）。
- `MM`：发布日期所在月份（2 位）。
- `total_commit_count`：发布 commit 时仓库的总 commit 数（`git rev-list --count HEAD`）。

示例：在 2026-06 发布且仓库总 commit 数为 87 时，版本号为 `2026.06.87`。
