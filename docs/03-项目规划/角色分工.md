# 角色分工

本项目适合 CCB 多 agent 协作。建议分工如下。

## 产品负责人 / 需求裁剪

职责：

- 判断产品方向是否符合“命令工作台”定位。
- 决定 MVP 范围。
- 评估交互体感。
- 决定命名。

关注问题：

- 是否仍然保留真实命令的可见性？
- 是否比直接查 help 更轻？
- 是否避免了重菜单化？
- 是否能让新手理解命令，也不妨碍熟手输入？

## 交互设计 agent

职责：

- 设计内容区 + 输入区布局。
- 设计 token 追加、删除、补全、候选选择。
- 设计 AI slash command 的交互。
- 设计执行结果展示方式。

交付物：

- TUI 信息架构。
- 键盘/鼠标交互说明。
- 原型截图或文本 mock。

## CLI 知识建模 agent

职责：

- 将外部 CLI 工具建模为命令树。
- 为每个节点补充标题、说明、风险、示例。
- 设计动态候选源。

第一阶段范围：SDKMAN。

交付物：

- SDKMAN command tree。
- Java/Maven/Gradle candidate 说明。
- Identifier 解析规则。

## 架构 agent

职责：

- 设计核心模型：CommandNode / CommandSession / CandidateProvider / Executor。
- 设计 i18n 结构。
- 选择 TUI 技术栈。
- 规划和旧 CUI 的关系。

交付物：

- 架构设计。
- 模块边界。
- 技术选型建议。

## 实现 agent

职责：

- 搭建项目骨架。
- 实现命令树加载。
- 实现 TUI 原型。
- 实现 SDKMAN 执行器。
- 实现 slash command。

要求：

- 每完成一个独立问题，单独提交 commit。
- 不把多个问题堆在一个 commit。
- 每次提交前跑相关测试。

## 测试 agent

职责：

- 编写单元测试。
- 编写交互 smoke test。
- 验证 SDKMAN 在 ARM64 Linux、macOS、x86_64 Linux 上的行为差异。
- 验证 shell function 命令执行。

测试重点：

- token 追加/删除。
- 命令树导航。
- 动态候选解析。
- slash command 解析。
- 命令执行失败后的 `/fix` 上下文。
- 多语言 fallback。

## 文档 agent

职责：

- 维护 README。
- 维护用户使用说明。
- 维护命令域扩展指南。
- 维护贡献指南。

## 推荐 CCB 协作流程

1. 产品负责人确认 MVP。
2. 架构 agent 给出模型和技术栈建议。
3. 交互设计 agent 给出文本原型。
4. CLI 知识建模 agent 输出 SDKMAN 命令树。
5. 实现 agent 做最小 prototype。
6. 测试 agent 验证体验和边界。
7. 产品负责人试用并反馈。
8. 进入下一轮循环。
