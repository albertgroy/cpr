# T-001 SDKMAN MVP 复盘（2026-06-19）

任务：T-001 — 命令构造器 SDKMAN MVP。
分支：`feat/T-001-sdkman-mvp`。
参与角色：archi（最终验收）、agent1 (claude)、agent2 (opencode)、ccb_self（协调者，本人）。
评估口径：基于事实证据（commit / artifact / 评审回复），不夸不糖衣。

## 1. archi（最终验收）— 亮眼

亮点：

1. 6/19 契约评审一次性点名 6 个 P1 缺口（id 正则、parse-link 两阶段、`param.source` 枚举、`unparsedTail` / 环形缓冲 N=5、错误结构、cancel 行为），没有放任何"先写写看"口子，避免后续契约漂移。
2. 终验识别**缺口 A**（DynamicCandidateProvider 未接入 TUI）。所有自动化测试通过、coverage 88%、agent1+agent2+协调者都未发现；archi 通过读 `cpr/tui/app.py:run_app` 的 `static_resolver` 注入路径直接抓到——场景 4 端到端走不通。这是评审者最该抓的那种实质问题。
3. 终验抓**缺口 B**（`scripts/manual_sdk_smoke.sh` 只测 SDKMAN 不测 CPR），属于流程层把关，协调者派单疏忽的兜底。
4. 补丁评审节奏控制：明确说"对照终验 §2/§3 检查即可，不再走全量评审"，节省一轮往返。
5. 4 项 P3 小改判定颗粒度合适：每条说明现状、影响、触发条件、修复方向，不是 nitpick，留了"什么时候做"的钩子。

拉胯点：无。

## 2. agent1（核心 + 测试 + 脚本，claude）— 达标偏稳

亮点：

1. core 层（model / session / candidates / executor / slash）一次过 archi 契约评审，未要求返工。
2. coverage 88.04% 一发到要求线以上（终验补丁后达到 89.31%）。
3. 单测覆盖 backspace / esc / 环形缓冲 / parse-link 两阶段 / executor 超时 / 安装版本枚举等关键路径。
4. dynamic 候选 fixture 与真实 SDKMAN 输出漂移问题被主动记入 `docs/03-项目规划/变更与缺陷队列.md` 作为 P2，没硬骗"已支持"。

拉胯点：

1. **`scripts/manual_sdk_smoke.sh` 只测 SDKMAN 不测 CPR**（终验缺口 B 根因）。agent1 写脚本时未回头确认"smoke 到底验收谁"。该要求来自 archi 6/19 评审 §5.2 草稿；协调者派单未显式翻译为验收命令清单是首责，但实现端可以多问一句。
2. `cpr/core/model.py:_parse_node` 在节点 dedup 时被调用两次（archi 终验次级问题 #6），微小 CPU 浪费 + 走查易误读为"重复 parse 是有意 dedup"。

## 3. agent2（TUI + YAML + README，opencode）— 达标，主动反馈到位

亮点：

1. prompt_toolkit 布局一次到位：输入 / 内容 / 候选 / 状态条四区分清，`_initial_content` 给中英双语欢迎信息，体感比 MVP 预期成熟。
2. 自我反馈写进回包："slash dispatch 未接入 run_app"、"`dynamic-pending` 不在 `param.source` 枚举内"——后者是契约符合性问题不是功能 bug，识别准确。比沉默交付强很多。
3. `tests/data/test_sdkman_yaml.py` 把 yaml 结构做断言，避免字段拼写漂移。

拉胯点：

1. **没把"占位标签 `dynamic-pending`"的因果链推到"场景 4 端到端缺失"**。`static_resolver` 对 param 节点返回空、占位标签存在，这两个事实 agent2 都知道，但只反馈了契约层（标签错），没反馈集成层（端到端没通）。这是 archi 终验缺口 A 协调者补的位置，agent2 本可以更早抛出来。
2. **第三个反馈"`--check-tree` / `--slash-parse` 入口缺失"是错的**——实际已在 `cpr/__main__.py` 提供。给协调者制造伪问题排查。**反映 agent2 没读 `__main__.py` 就提反馈。** 小问题但是属于"反馈应基于事实"。

## 4. ccb_self（协调者，本人）— 整体合格，2 个明显失误

亮点：

1. 启动期 4 问 4 答（语言/git/派单顺序/命名）决策落地到 `docs/04-会议记录/2026-06-19-启动决策.md`，无后悔门。
2. 严格按"契约 → 派单 → 验收 → archi 评审"环走，无跳步。
3. integration commit `660f46b` 主动闭环 agent2 两个开放问题（slash dispatch、dynamic-pending → dynamic），未把噪音留给 archi。
4. 终验阻塞项 A/B 用 3 个补丁 + 1 个文档修订一次性收口（`1346543` / `a1ef169` / `6cbe9da` / `e967713`），无反复拉锯。
5. 每个补丁配对应单测（composite resolver 4 例 + slash dispatch 4 例），不裸改代码。

拉胯点（诚实承认）：

1. **派 agent1 写 smoke 脚本时未明确"必须经过 `python -m cpr`"。** archi 6/19 评审 §5.2 草稿写过这个要求，派单时未显式翻译为验收命令清单——结果脚本是裸 SDKMAN 自检。**协调者派单质量问题。**
2. **integration 阶段错过缺口 A。** `660f46b` 改 slash dispatch + `dynamic-pending → dynamic` 时，没问"用户输 `sdk install java` 进 TUI，候选区会显示什么"。当时随手 mock executor 试一下就能发现 `static_resolver` 对 param 节点返空。**把 integration 当作"修标签"做了，没当作"端到端走一遍"做——这次任务最该自责的失误。**
3. 第一版 `run_app.execute` 用 `state.session.record_result(...)` + 直设 `last_command`——实际 API 是 `record_execution(command, SessionResult)`。**ccb_self 未读 `cpr/core/session.py` 直接写代码的小坑**，后续读源码才修正。

## 5. 团队层观察

1. **第二开发对（agent3 codex + agent4 kiro-cli）未启用**。文件边界对 agent1+agent2 够清晰，MVP 量级任务这是对的判断，无"为了用满而硬塞"。
2. **CCB 通信稳定**。唯一一次 `ask --callback` 被拒（"requires an active parent job"）是 ccb_self 对 CCB 语义不熟，立刻改 plain `ask` 解决，无后续卡顿。
3. **缺口 A 暴露团队系统性问题**：integration / 验收阶段无人做端到端 mock 验证。agent1 跑单测、agent2 跑 TUI 启动、协调者做集成、archi 看代码——四个人都没在"用户在 TUI 输 `sdk install java`"这个具体场景上 mock 跑通。**下次类似 MVP 任务建议**：派单时强制要求"集成阶段必须有一个 e2e mock 测试覆盖契约场景"，纳入 acceptance 验收口径。

## 6. 一句话总结

archi 这次扛起了真正的把关角色，agent1 / agent2 完成度都达标但都缺最后那一步"跨边界检查"，协调者派单和集成两阶段各漏了一次端到端意识——好在 archi 兜住了。
